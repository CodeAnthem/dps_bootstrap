# Remote unlock (initrd SSH)

Unlock a LUKS-encrypted machine over the network at boot, before the disk is
decrypted. NDS starts a tiny SSH server **in the initrd**; you SSH in and enter
the LUKS passphrase remotely, so the box boots without a physical keyboard.

This is configured by **classicInstall** when you enable
**encryption -> remote unlock**.

## How it works

At boot, before decryption:

- Initrd SSH listens on **port 22**, login user **root**, **pubkey-only**.
- The authorized key is the public key you provide during configuration.
- The initrd host key is generated by NDS and shipped in the backup bundle at
  `secrets/initrd_ssh_host_ed25519_key`.
- Once the NIC is up, NDS prints the address and a connect hint to the **console**
  (so you can see the IP even on a fresh DHCP lease).

Logging in and unlocking are **two steps** rolled into one: the key gets you into
the initrd, and the authorized key is wired with `command="systemctl default"`,
so the connection drops you **straight into the passphrase prompt**.

## The two keys (don't mix them up)

| Key | Who owns it | Job |
|-----|-------------|-----|
| **Host key** | The NixOS machine | Identifies the initrd to your SSH client. NDS generates it automatically; it's in the bundle at `secrets/initrd_ssh_host_ed25519_key`. |
| **Authorized key** | **You** (your PC) | Authenticates *you*. You paste your PC's **public** key into the "Authorized SSH public key" prompt; only the matching **private** key can log in. |

## Make an unlock key on your PC

Use a **dedicated** key for unlocking (not your everyday `id_ed25519`). The
canonical name in these docs is **`nixos-unlock`**.

**Linux / macOS / WSL:**

```bash
ssh-keygen -t ed25519 -f ~/.ssh/nixos-unlock
cat ~/.ssh/nixos-unlock.pub
```

**Windows (cmd.exe / PowerShell):** use `%USERPROFILE%` and backslashes - `~/`
does not work here (see the note below).

```bat
ssh-keygen -t ed25519 -f "%USERPROFILE%\.ssh\nixos-unlock"
type "%USERPROFILE%\.ssh\nixos-unlock.pub"
```

Leave the passphrase empty for unattended use, or set one for extra safety.

The `.pub` line looks like `ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... you@yourpc`.
Paste **that full line** into the "Authorized SSH public key" prompt.

> The `SHA256:...` line ssh-keygen prints is only a **fingerprint** (a hash for
> visual ID) - it is **not** the key and will not work.

### Why `~/` works for `ssh` but not `ssh-keygen`/`type` on Windows

`cmd.exe` and PowerShell do **not** expand `~` - that's a Unix-shell feature. It
looks inconsistent because:

- `ssh -i ~/.ssh/nixos-unlock ...` **works**: the OpenSSH **client** expands a
  leading `~/` in its identity-file argument itself.
- `ssh-keygen -f ~/.ssh/nixos-unlock` **fails**: ssh-keygen writes the `-f` path
  **literally**, so it tries to create a folder actually named `~` and reports
  `No such file or directory`.
- `type ~/.ssh/...` **fails**: `type` is a shell builtin that never expands `~`.

On Windows, always pass a real path: `"%USERPROFILE%\.ssh\nixos-unlock"`.

### Reuse vs. per-machine

- **One dedicated unlock key reused across your machines** is the practical sweet
  spot for a homelab/small fleet - easy to manage, still isolated from your
  everyday key.
- Go **per-machine** only when boxes live in different trust zones or you need to
  revoke one without touching the others.
- Add **each admin workstation's** public key you might connect *from*, so losing
  one laptop doesn't lock you out.

## Connect and unlock

When the machine reaches the LUKS prompt (watch the console for the address):

```bash
ssh -i ~/.ssh/nixos-unlock root@<machine-ip>
# Windows also accepts:  ssh -i "%USERPROFILE%\.ssh\nixos-unlock" root@<machine-ip>
# the passphrase prompt appears automatically; type it and the machine boots
```

Login is as **root** in a minimal early-boot environment - not your admin user,
which only exists once the real system has booted.

## Finding the IP

- **Static IP** (recommended): always reachable at exactly the address you set.
- **DHCP**: NDS makes the initrd request its lease by **MAC address**, the same
  identity the booted system uses, so the initrd gets the **same** IP as the
  running machine. NDS also prints the address on the console at boot. If your
  DHCP server reassigns, check its lease logs.

## Quirks

### Host-key change warning (same IP, two SSH servers)

The initrd sshd and the booted sshd share the machine's IP but have **different
host keys**, so your client may warn about a changed host key when you switch
between unlocking and normal SSH. This is expected. Options: keep a separate
`known_hosts` entry, use a `HostKeyAlias` in `~/.ssh/config` for the initrd, or
just accept the new key when prompted.

### "Failed to connect to system scope bus via local transport"

Harmless. When the forced `systemctl default` runs in the initrd it prints a
D-Bus notice, then the passphrase prompt appears normally and unlocking
proceeds. It is not an error.

## Recovery / if your PC dies

The SSH key is a **convenience**; the **LUKS passphrase** (in the backup bundle)
is the real safety net - you can always unlock at the physical/VM console by
typing it directly.

- **Back up your private key** (`~/.ssh/nixos-unlock`) somewhere safe - encrypted
  USB or a password manager. Restore it on a new PC and you're back in.
- **Authorize a second machine** by adding more keys to
  `boot.initrd.network.ssh.authorizedKeys` in the generated config (or reinstall
  with a different key).
